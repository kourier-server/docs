Writing Well-Behaved Peers

RPC frameworks boost productivity by abstracting from developers everything related to the network. But, if not chosen wisely, the provided abstractions can make it impossible to implement well-behaved peers or deal with malicious ones. That's why developers commonly use RPC frameworks behind a firewall in enterprise deployments or controlled, non-publicly available, microservices-based environments protected behind API Gateways or Application Load Balancers/Firewalls.

Well-designed RPC frameworks choose abstractions wisely to promote productiveness without depriving developers of the ability to deal with malicious peers or write well-behaved ones.

Cute is a true evolution in this regard. Unlike many RPC frameworks, Cute boosts developers' productivity by using signals and slots as a solid foundation for the interaction model it provides and that smartly abstracts network communication without hiding from developers important network behaviors that they must be aware of to write reliable and secure code.

Monitoring how fast peers are sending data on the network is an all-important behavior that developers must have access to. Cute abstracts client-server interaction through the RemoteObject class it provides on both Cute::Client and Cute::Server namespaces. Both of these classes have the following methods to aid developers in monitoring and controlling data writing on the network:

// The methods below exist both on
// Cute::Client::RemoteObject and
// Cute::Server::RemoteObject.
public:
qint64 bytesToWrite() const;
signals:
void bytesWritten();

Cute uses WebSocket connections for remote object interaction and shares them with remote objects. Whenever the underlying TCP socket of the WebSocket connection sends data to the connected peer, all remote objects sharing the WebSocket connection emit the bytesWritten signal. When handling this signal, developers can use the bytesToWrite method to know if there is more data to be written by the socket, meaning that the RemoteObject instance will emit the bytesWritten signal again at a later time, and how much data is still pending to be written by the socket, so they can opt to send further data when the bytesWritten signal gets emitted again. Unlike many other RPC frameworks, Cute empowers developers to control how much data is sent to the connected peer, enabling them to write well-behaved peers quickly and easily.
